# TS-0060 — Startup Reconciliation

**Phase:** 4 — Reconciliation (depends on TS-0050)  
**Status:** Pending

---

## Goal

On startup, detect any order state changes that occurred while the program was offline (e.g., a pending entry order that filled, a GTC TP that triggered). `reconcile()` reloads local DB state, fetches fresh broker data, diffs them, replays missed events chronologically, and logs a reconciliation summary. Also wire `reconcile()` into the `poll` CLI command so it runs automatically before the loop starts.

This story answers the FAQ question: *"We reboot the local computer — during the down time the trade fills. How can the system figure out that the order filled and update its status?"*

## Design

### `reconcile(conn, client, account_id) -> ReconcileResult`

```python
@dataclass
class ReconcileResult:
    checked: int      # total orders fetched from broker
    replayed: int     # events emitted for missed transitions
    summary: str      # human-readable one-liner
```

Steps:

1. Fetch all current orders from broker → `fresh: list[Order]`
2. For each fresh order, call `detect_events(fresh=[order], get_known=lambda oid: get_order(conn, oid))` — reuses the same pure diff logic from `engine.py`
3. For each detected event:
   a. Upsert the order
   b. Append an `EventLog` entry (details include `"reconciled": true`)
   c. If `EventType.filled` → call `assign_trade(conn, order)`
4. Update `poll_state.last_poll_at`
5. Return `ReconcileResult`

### Integration in `poll` CLI command

```python
result = reconcile(conn, client, cfg.account_id)
console.print(f"[bold]Reconciliation:[/bold] {result.summary}")
run_poll_loop(...)
```

|---|---|

## Notes

- `reconcile()` is intentionally thin — it delegates all event logic to `detect_events` and `assign_trade` already covered by their own tests. This test suite only validates the reconciliation-specific behaviour (the `"reconciled"` flag, the summary, the startup wiring).
- If the broker returns orders not in DB (new orders placed via desktop while offline), they are treated as `new_order` events and upserted — but no Trade # is assigned since no fill has been confirmed. The engine will assign a Trade # on the next `filled` event.
