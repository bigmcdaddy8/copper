## Goal

Create `engine.py` — the core polling loop that fetches broker data, diffs it against the last known DB state, detects order-level events, persists them to `event_log`, and updates the `orders` table. Also wire up the `tradier_sniffer poll` CLI sub-command that runs the loop.

This story answers the FAQ question: *"Is the API order placed asynchronously — can we poll?"* (yes — confirmed by implementation).

## Design

### Event detection logic

On each poll cycle, `engine.poll()`:

1. Calls `tradier_client.get_orders(account_id)` → list of raw order dicts
2. For each raw order, builds an `Order` dataclass
3. Compares against the last known `Order` from DB (`get_order`):
   - **New order** — `order_id` not in DB → emit `EventType.new_order`, upsert
   - **Status change** — `order_id` in DB, `status` differs → emit matching event type, upsert
     - `filled` or `partially_filled` → `EventType.filled`
     - `canceled` or `rejected` or `expired` → `EventType.canceled`
4. Upserts every fetched order (refreshes `updated_at`, `fill_price`, etc.)
5. Updates `poll_state.last_poll_at` to current UTC timestamp

### `detect_events` helper

```python
def detect_events(
    fresh: list[Order],
    get_known: Callable[[str], Order | None],
) -> list[tuple[Order, EventType]]:
```

Pure function — no I/O, no DB calls. Takes the fresh order list and a lookup callable; returns `(order, event_type)` pairs for orders that warrant an event. Fully unit-testable without a DB or HTTP client.

### `poll()` function

```python
def poll(conn: sqlite3.Connection, client: TradierClient, account_id: str) -> list[EventLog]:
```

Executes one poll cycle. Returns the list of `EventLog` entries that were appended. Callers (the CLI loop and tests) can inspect what happened.

### `run_poll_loop()` function

```python
def run_poll_loop(conn: sqlite3.Connection, client: TradierClient, account_id: str, interval: int) -> None:
```

Calls `poll()` in a `while True` loop with `time.sleep(interval)` between cycles. Catches `KeyboardInterrupt` cleanly (prints "Polling stopped." and exits). Prints a one-line Rich status after each cycle showing timestamp, orders checked, and events detected.

### `tradier_sniffer poll` CLI sub-command

```
tradier_sniffer poll [--interval SECONDS]
```

- Default interval from `SnifferConfig.poll_interval` (10 s)
- Calls `run_poll_loop()`
- Opens DB connection via `init_db(config.db_path)`

## Tasks

### 1. Implement `engine.py`

- `_raw_to_order(raw: dict, account_id: str) -> Order` — converts a Tradier order dict to an `Order` dataclass
  - Maps `raw["class"]` → `Order.class_`; handles missing/null fields defensively
  - `status` field → `OrderStatus` enum; unknown values default to `OrderStatus.pending`
- `detect_events(fresh, get_known)` — pure diff logic (see above)
- `poll(conn, client, account_id)` — one cycle: fetch → diff → persist → update poll_state
- `run_poll_loop(conn, client, account_id, interval)` — loop with sleep + Rich output

### 2. Add `poll` sub-command to `cli.py`

```python
@app.command()
def poll(ctx: typer.Context, interval: int = typer.Option(None, help="Seconds between polls")):
```

Uses `ctx.obj` (the `SnifferConfig`) for `account_id`, `db_path`, and default `poll_interval`.

### 3. Implement `tests/test_engine.py`

All tests use in-memory DB from `init_db(":memory:")` and a simple stub for `get_known`.

## Notes

- Multi-leg fill detection: Tradier returns a multileg order as a single order object with legs. A fill on the parent order means all legs are filled. `detect_events` treats the parent `order_id` as the unit of detection — per-leg fill prices are captured in `legs_json` via `upsert_order`.
- Partial fills (`partially_filled`) are emitted as `EventType.filled` events since they represent real capital deployment; the engine re-emits if status later changes to `filled`.
- `run_poll_loop` does not handle reconnection or client errors beyond logging — error handling is deferred to a future story.
- The `poll` command is intentionally blocking; background execution is a Phase 6 concern.
