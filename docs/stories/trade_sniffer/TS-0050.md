# TS-0050 — Trade # Assignment & Order Mapping

**Phase:** 3 — Core Engine (depends on TS-0040)  
**Status:** Pending

---

## Goal

Implement `assign_trade()` — the function that takes a filled `Order`, determines whether it belongs to an existing open `Trade` or needs a new one, creates the `Trade` record if needed, and writes the `TradeOrderMap` link. Also define the `TradeType` enum and integrate `assign_trade()` into the poll loop so that every `filled` event immediately gets a Trade # assigned.

This story answers the FAQ question: *"Does the brokerage have a reference # / tagging capability to map orders to trades?"* — documented as a finding in `TRADIER_FAQ.md` after sandbox exercise.

## Design

### Trade ID format

```
TRDS_{seq:05d}_{trade_type}
```

e.g. `TRDS_00001_NPUT`, `TRDS_00042_SIC`

`build_trade_id(seq: int, trade_type: TradeType) -> str` — pure formatting helper.

### Trade type inference (`infer_trade_type`)

Since Tradier has no semantic trade-type field, the trade type is inferred from the order's legs and side:

| Condition | Inferred type |
|---|---|
| Single-leg, `sell_to_open`, option | `NPUT` if put, `NCALL` if call (from OCC symbol) |
| Single-leg, `buy_to_open`, equity | `STOCK` |
| Single-leg, covered call (`sell_to_open`, option + existing long stock position) | `CCALL` — **cannot be inferred from order alone; caller passes explicit type** |
| Two legs, both options, different strikes, same expiry, puts | `PCS` |
| Two legs, both options, different strikes, same expiry, calls | `CCS` |
| Four legs (iron condor) | `SIC` |
| Anything else | `NPUT` as safe fallback |

`infer_trade_type(order: Order) -> TradeType` — pure function, no DB calls. Callers (demo scenarios) may override by passing `trade_type` explicitly to `assign_trade()`.

### Grouping logic (`assign_trade`)

```python
def assign_trade(
    conn: sqlite3.Connection,
    order: Order,
    trade_type: TradeType | None = None,
    role: str = "entry",
    opened_at: str | None = None,
) -> Trade:
```

1. If `order.tag` is non-empty: look up existing open `Trade` whose `trade_id` ends with the tag suffix — this handles broker-tagged orders (TS-0015 established Tradier supports a `tag` field).
2. Otherwise: look for an open `Trade` linked to an order with the same `symbol` and a `mapped_at` within 5 seconds of `order.created_at` — proximity grouping for multileg orders placed together.
3. If no existing trade found: create a new one via `next_trade_sequence` + `build_trade_id`.
4. Write `TradeOrderMap(trade_id, order.order_id, role, now)`.
5. Return the `Trade`.

`opened_at` defaults to `order.created_at` if not supplied.

## Notes

- OCC symbol format: `{underlying}{YY}{MM}{DD}{C|P}{strike*1000, 8 digits}` — the 7th character from the end of the root (after stripping strike) is `C` or `P`. For practical parsing: find the first `C` or `P` after the date portion. The regex `[A-Z]+(\d{6})([CP])` on the symbol suffix is sufficient.
- Proximity grouping (5-second window) is intentionally simple — in practice multi-leg orders arrive within milliseconds of each other. The window can be tuned in TS-0080 if needed.
- `assign_trade()` is idempotent with respect to the `TradeOrderMap` — `insert_trade_order_map` uses `INSERT OR IGNORE`.
