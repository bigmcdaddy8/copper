# K9-0110 — Error Taxonomy + Actionable Failures

**Phase:** 7 — Reliability & YAML Cutover
**Status:** Completed
**Depends on:** K9-0090 (End-to-End Integration + Guard Rails), K9-0100 (README & Example Trade Specs)

## Goal

Replace opaque runtime failures with stable, actionable errors surfaced consistently in CLI output and run logs.

Current runs can fail with low-context messages (for example, raw key/time lookup errors). This story introduces a small error taxonomy and maps common failure points to predictable error codes/messages so operators can identify root cause quickly.

## Tasks

### 1. Define error categories

Add a small, internal error taxonomy for MVP operations, e.g.:

- `DATA_UNAVAILABLE`
- `SELECTION_FAILED`
- `ORDER_REJECTED`
- `BROKER_ERROR`
- `CONFIG_ERROR`

Represent this as constants or a light enum; avoid broad architectural churn.

### 2. Map runner failures to categories

In `runner.py`, catch and classify known failures around:

- quote/chain retrieval
- strike selection
- order placement/polling
- TP placement

Populate `RunResult.reason` with operator-friendly text and populate `errors` with machine-readable strings including the category.

### 3. Surface category in logs and CLI

Ensure `RunLog.record()` persists category fields and CLI output displays concise actionable text.

### 4. Add tests

Add focused tests that simulate known exceptions and assert:

- outcome is `ERROR`
- reason is actionable
- category/code appears in result/log/CLI text
